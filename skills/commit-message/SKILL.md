# Commit Message

根据 Git 改动生成简体中文 Conventional Commit。默认只生成提交信息；只有用户明确要求提交时才执行 `git commit`。

## 工作流

先读取 `git status --short`、`git diff --cached --stat` 和 `git diff --cached`。暂存区有内容时，以暂存区为唯一依据；未暂存和未跟踪文件只作为风险提醒，不纳入提交信息。

暂存区为空时，按用户意图处理：如果用户只是要求生成提交信息，可以读取 `git diff --stat` 和 `git diff`；发现 `??` 未跟踪文件时，先读取文件内容或确认是否纳入分析，并说明该信息基于未暂存改动。如果用户要求执行提交，先确认要暂存哪些文件。只有用户明确说“全部提交”“提交所有改动”等范围清晰的话，才可以暂存全部当前改动。

检查 diff 是否包含多个独立意图。能拆分时，建议拆分提交并说明候选提交边界；用户要求继续合并时，选择主导意图作为 header，其余重要改动放入 body。

生成提交信息后再次检查：type 是否准确，scope 是否必要，subject 是否精简，body/footer 是否只在需要时出现，且没有任何 AI 署名或工具痕迹。

执行提交时，使用生成的信息运行 `git commit`。多行提交信息优先用临时文件配合 `git commit -F`，临时文件放在仓库外的临时目录并在提交后删除。不要使用 `--no-verify`，不要自动推送。

## 输出规则

用户只要求提交信息时，只输出提交信息纯文本，不加代码块，不解释推理过程。用户要求执行提交时，提交成功后简要报告生成的提交信息和 commit hash；失败时报告失败原因与下一步。

提交信息格式：

```text
<type>(<scope>): <subject>

<body>

<footer>
```

只有 header 必需。没有 body 或 footer 时，连同相邻空行一起省略。

## Header

type 必须从下列值中选择：

| type | 用途 |
| --- | --- |
| feat | 新增用户可感知功能 |
| fix | 修复缺陷或错误行为 |
| refactor | 调整内部结构且不改变外部行为 |
| perf | 带来明确性能收益 |
| style | 纯格式、空白、分号、lint 等无语义变化 |
| docs | 文档变更 |
| test | 测试新增或调整 |
| build | 构建系统、依赖管理、打包配置 |
| ci | CI/CD 配置 |
| chore | 不影响产品行为的杂务 |
| revert | 回滚提交 |

scope 可选。优先使用组件名、页面/模块名、目录名或明确的业务域；跨多个无共同边界的模块时省略。不要使用 `misc`、`common`、`update` 这类笼统 scope。

subject 使用简体中文动宾结构，祈使语气，不超过 50 个中文字符，结尾不加标点。避免“了”“的”“的问题”“进行”“为了”“来”等冗余词。

## Body 与 Footer

简单改动只写 header，不要为了凑格式生成 body。改动跨多文件、逻辑较复杂、包含迁移步骤，或 subject 无法覆盖关键行为时，才生成 body。body 用 `-` 分点，每条单行陈述「做了什么」以及必要的动机或影响，与 subject 不重复；3–5 条为宜，超过说明应拆分提交。禁止按文件罗列改动，也不要逐行翻译 diff。

每个 bullet 必须独占一行，使用真实换行符分隔，禁止把多条 bullet 拼接在同一行内，也禁止输出 `\n`、`\\n` 这类转义字面量来代替换行。生成多行提交信息时务必通过 `git commit -F <文件>` 写入，不要用 `git commit -m` 拼接含转义符的字符串。

存在明确不兼容变更时，footer 使用 `BREAKING CHANGE:` 开头，并写清影响与迁移方式。只有用户、分支名、diff 或上下文明确提供 issue 编号时，才添加 `Closes #<ID>`。

回滚提交使用固定格式：

```text
revert: <被回滚的原 subject>

This reverts commit <hash>.
```

如果无法确定被回滚提交的 hash，先向用户确认，不要猜测。

## 禁止内容

提交信息不得包含任何 AI 智能体、工具或平台署名，包括 `Co-Authored-By`、`Co-authored-by`、`Signed-off-by`、`Generated by`、`Powered by`、`Assisted by`、AI 工具链接、bot 链接、`AI-generated` 或 emoji 水印。除非用户明确要求，不要添加任何 trailer。

## 示例

```text
style(Button): 调整边框颜色
```

```text
docs: 更新部署文档
```

```text
refactor(Auth): 重构登录状态管理

- 将 token 存储迁移到 Pinia
- 统一 Axios 响应拦截器的登出处理
- 移除冗余本地存储辅助函数
```

```text
fix(utils): 修复日期格式化异常

Closes #1024
```

```text
feat(api): 统一接口响应结构

BREAKING CHANGE: 响应体从 { data, code } 改为 { data, status, message }，调用方需要适配字段读取逻辑
```
